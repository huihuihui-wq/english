# 大文件上传优化方案（修正版）— 视频画面必需时的方案

> **分析对象**：Shadow Reader 本地音视频上传转录流程  
> **核心约束**：视频画面必须保留，播放器需要视频+音频同步播放  
> **分析日期**：2026-07-03

---

## 一、关键发现：视频播放其实已经是本地播放了！

从 `WelcomeScreen.tsx:123` 的代码：

```typescript
const videoUrl = URL.createObjectURL(pendingFile);
```

**这说明前端已经通过 `URL.createObjectURL` 创建了本地 blob URL 来播放视频。** 视频数据根本不需要离开浏览器，已经在本地播放了。

但紧接着 `handleUpload` 的代码：

```typescript
const result: TranscribeResponse = await transcribeFile(pendingFile, language, ...);
```

**又把整个文件上传到了后端做 ASR 转录。** 这才是问题所在。

---

## 二、当前链路的真实浪费

```
用户选择 2GB 电影文件
        ├─→ 前端 URL.createObjectURL(file) → 视频本地播放 ✅ 正确
        │
        └─→ 前端 transcribeFile(file) → 上传 2GB 到后端 ❌ 浪费
                    ↓
            【后端】await file.read() → Python 内存 2GB
                    ↓
            【后端】tmp_path.write_bytes() → 磁盘写入 2GB
                    ↓
            【ASR】_extract_audio_to_wav(file_bytes) → 再写 2GB 临时文件
                    ↓
            FFmpeg 提取 300MB 音频 → 丢弃 1.7GB 视频数据
                    ↓
            返回字幕
```

**视频数据被上传了、写入磁盘了、又传给 ASR 了，但 ASR 只提取了 300MB 音频。**

```
2GB 电影文件
├── 视频轨道 (H.264/HEVC)     ~ 1.7GB  (85%) ──→ 被上传了，但完全没用！
├── 音频轨道 (AAC/AC3/DTS)    ~ 0.3GB  (15%) ──→ ASR 只需要这部分
└── 字幕轨道                  ~ 几 MB  (0.1%)
```

---

## 三、第一性原理重新分析

既然视频已经在前端本地播放了，**问题不是"如何传视频"，而是"如何只传音频做 ASR"**。

核心问题变成：**前端浏览器没有 FFmpeg，怎么从 File 对象中提取音频轨道？**

---

## 四、修正后的方案对比

### 方案 A：前端 FFmpeg.wasm 提取音频（最优）

```
用户选择 2GB 电影
        ├─→ 前端 URL.createObjectURL(file) → 视频本地播放（零上传）✅
        │
        └─→ 前端 FFmpeg.wasm 提取音频轨道 → 200MB AAC
                    ↓
            上传 200MB 音频到后端
                    ↓
            【后端】ASR 转录 → 返回字幕
```

| 优点 | 缺点 |
|------|------|
| 视频播放零上传（blob URL） | FFmpeg.wasm 包体约 25MB，首次加载慢 |
| ASR 上传数据减少 90%（2GB→200MB） | 浏览器内存占用大（WASM 需要加载文件） |
| 后端无需改动 | 老旧设备可能不支持 WASM |
| 用户体验最好 | 集成复杂度中等 |

**适用场景**：追求极致体验，愿意接受前端集成复杂度

---

### 方案 B：后端 Streaming + FFmpeg 提取（工程最简）

```
用户选择 2GB 电影
        ├─→ 前端 URL.createObjectURL(file) → 视频本地播放（零上传）✅
        │
        └─→ 前端上传整个文件到后端（流式）
                    ↓
            【后端】UploadFile 逐块读取（不一次性入内存）
                    ↓
            接收完成后 FFmpeg 提取音频 → 丢弃视频
                    ↓
            ASR 转录 → 返回字幕
```

| 优点 | 缺点 |
|------|------|
| 前端完全无需改动 | 仍然上传了 2GB 视频数据（虽然画面本地播放了） |
| 后端内存恒定（不随文件增长） | 上传时间较长（对于本机回环来说 2GB 约 5-10 秒） |
| 支持任意大小文件 | 逻辑上很蠢：视频已经本地播放了，又传了一遍 |
| 实现复杂度最低 | |

**适用场景**：快速落地，前端不改，优先解决"不能传大文件"的问题

---

### 方案 C：后端直接读取本地文件路径（理论最优，需桌面端）

```
用户选择 2GB 电影
        ├─→ 前端 URL.createObjectURL(file) → 视频本地播放
        │
        └─→ 前端获取文件绝对路径（C:\Movies\xxx.mkv）→ 传路径给后端
                    ↓
            【后端】直接 ffmpeg -i C:\Movies\xxx.mkv → 提取音频
                    ↓
            ASR 转录 → 返回字幕
```

| 优点 | 缺点 |
|------|------|
| 零数据传输（只传路径字符串） | **浏览器安全限制：无法获取文件真实路径** |
| 内存占用最小 | 需要 Electron/Tauri 桌面端才能实现 |
| 速度最快 | |

**适用场景**：如果未来打包为桌面应用（Electron/Tauri），这是最佳方案

---

## 五、关键决策

| 决策 | 建议 | 理由 |
|------|------|------|
| 视频播放方式 | **保持 blob URL 本地播放** | 已经是正确的做法，不需要改动 |
| 是否去掉 200MB 限制？ | **是** | 本地应用不应有比用户磁盘更小的限制 |
| 前端是否做 FFmpeg.wasm？ | **Phase 2** | 效果显著但集成成本高，Phase 1 先保底 |
| 后端是否做 Streaming？ | **Phase 1** | 前端不改，立即支持大文件，内存不再随文件增长 |
| 是否支持桌面端路径？ | **未来** | 需要 Electron/Tauri 改造，不在当前 scope |

## 六、Phase 1 详细设计：后端 Streaming + 音频提取

### 6.1 当前代码问题（`main.py:364-377`）

```python
file_bytes = await file.read()          # 2GB 读入 Python 内存！
# ...
tmp_path.write_bytes(file_bytes)        # 2GB 写入磁盘
# ...
await transcribe_audio(file_bytes, ...) # 2GB 传给 ASR
```

### 6.2 改造后流程

```python
# 1. 流式接收文件到临时位置（内存恒定）
chunk_size = 8192
with open(tmp_video_path, "wb") as f:
    while chunk := await file.read(chunk_size):
        f.write(chunk)                  # 每次只处理 8KB

# 2. FFmpeg 提取音频（丢弃视频数据）
ffmpeg -i tmp_video.mp4 -vn -acodec libmp3lame -ac 1 -ar 16000 -b:a 64k tmp_audio.mp3

# 3. 只把音频传给 ASR（可能只有 50-300MB）
audio_bytes = tmp_audio_path.read_bytes()
await transcribe_audio(audio_bytes, "audio.mp3", "audio/mpeg", language)

# 4. 清理临时文件
os.unlink(tmp_video_path)  # 删除 2GB 视频临时文件
os.unlink(tmp_audio_path)  # 删除音频临时文件
```

### 6.3 前端改动

**Phase 1 前端无需改动。** 视频播放已经是 `URL.createObjectURL` 本地播放，上传部分保持 `transcribeFile` 不变。

可选：添加大文件提示文案（"大文件上传可能需要一些时间，请耐心等待"）。

## 七、Phase 2 设计：前端 FFmpeg.wasm 提取音频

```
用户选择 2GB 电影
        ├─→ 前端 URL.createObjectURL(file) → 视频本地播放（零上传）
        │
        └─→ 前端 FFmpeg.wasm 提取音频
                    ↓
            2GB → 200MB 音频（AAC/MP3）
                    ↓
            上传 200MB 到后端
                    ↓
            【后端】直接 ASR 转录 → 返回字幕
```

**前端改动**：
1. 集成 FFmpeg.wasm（约 25MB 包体）
2. 用户选择文件后，先提取音频，再上传音频
3. 上传进度显示"提取音频中..." → "上传音频中..."

**后端改动**：接受纯音频文件，无需视频提取步骤

## 八、Phase 3 设计：桌面端直接路径

如果用 Electron/Tauri 打包：

```python
# 后端直接读取本地文件
@app.post("/api/transcribe")
async def transcribe(file_path: str = Form(...), language: str = Form("en")):
    # 直接读取本地文件，无需上传
    audio_bytes = extract_audio_from_path(file_path)

```

**零数据传输，最优雅。**

## 九、效果预期

| 指标 | 当前 | Phase 1 后 | Phase 2 后 | Phase 3 后 |

| 视频播放 | 本地 blob URL ✅ | 本地 blob URL ✅ | 本地 blob URL ✅ | file:// 协议 ✅ |

---

**结论**：视频播放已经正确实现（`URL.createObjectURL` 本地播放），核心问题是上传 ASR 时传了无用的视频数据。Phase 1（后端 Streaming）是最务实的下一步，前端不改，后端内存不再随文件增长。Phase 2（FFmpeg.wasm）是体验优化的下一步，可减少 90% 上传数据。Phase 3（桌面端）是终极方案，零数据传输。
