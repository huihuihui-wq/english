# 大文件上传优化方案 — 第一性原理分析汇报

> **分析对象**：Shadow Reader 本地音视频上传转录流程  
> **核心问题**：不支持几百 MB~几 GB 的电影/视频文件上传  
> **分析日期**：2026-07-03

---

## 一、当前上传链路全景（问题定位）

```
用户选择 2GB 电影文件
        ↓
【前端】FileReader → 浏览器内存 2GB
        ↓
【网络】XMLHttpRequest POST 到 localhost:8000/api/transcribe
        ↓
【后端】await file.read() → Python 内存 2GB
        ↓
【后端】tmp_path.write_bytes() → 磁盘写入 2GB
        ↓
【ASR】_extract_audio_to_wav(file_bytes) → 临时文件 2GB → FFmpeg 提取音频
        ↓
生成字幕
```

### 1.1 瓶颈清单

| 环节 | 当前行为 | 问题 | 2GB 电影影响 |
|------|---------|------|-------------|
| 上传限制 | `MAX_UPLOAD_MB = 200` | 直接拒绝大文件 | 2GB 文件被 413 拒绝 |
| 前端内存 | 整个文件读入浏览器内存 | 大文件导致浏览器卡死/崩溃 | 2GB 超出浏览器内存限制 |
| 网络传输 | HTTP POST 原始文件 | 传输 2GB 数据到本机后端 | 完全冗余（前后端在同一台机器） |
| 后端内存 | `await file.read()` 读入 bytes | 2GB 同时驻留 Python 内存 | 可能导致 OOM |
| 磁盘 IO | 先写临时文件再处理 | 2GB 写入 + 2GB 读取 | SSD 寿命和速度瓶颈 |
| ASR 输入 | 把整个文件传给 `_extract_audio_to_wav` | 处理了 1.7GB 无用的视频画面数据 | ASR 最终只提取音频，但原始文件已完整读入 |

**关键发现**：后端 ASR 已经会用 FFmpeg 提取音频（`-vn -map 0:a`），但原始文件在到达 ASR 之前已经被完整读入内存和写入磁盘了。

## 二、第一性原理分析

### 2.1 我们真正需要什么？

学英语影子跟读，核心数据只需要两样：
1. **音频轨道** → 给 ASR 转录成文本/时间轴
2. **字幕文本** → 展示给用户对照

### 2.2 2GB 电影文件的构成

```
2GB 电影文件 (.mkv/.mp4)
├── 视频轨道 (H.264/HEVC)     ~ 1.7GB  (85%)
├── 音频轨道 (AAC/AC3/DTS)    ~ 0.3GB  (15%)
└── 字幕轨道 (PGS/ASS/SRT)    ~ 几 MB  (0.1%)
         ↑
    ASR 只需要这部分
```

**视频画面数据占总文件的 85%，对 ASR 完全无用。**

### 2.3 本地部署的特殊性

Shadow Reader 是**本地运行**的应用：
- 前端：`http://localhost:5173`
- 后端：`http://localhost:8000`
- 用户文件：本地磁盘

这意味着文件"上传"实际上是这样走的：

```
本地磁盘 → 浏览器内存 → HTTP/TCP 回环 → Python 内存 → 本地磁盘
```

**同一台机器上的文件绕了一圈网络，这是完全冗余的。**

### 2.4 ASR 真正需要什么格式？

faster-whisper（后端 ASR 引擎）的最佳输入：
- **16 kHz 采样率**
- **单声道**
- **WAV 或 MP3 格式**

2GB 电影的音频轨道（假设 2 小时）：
- 原始 AAC 立体声 48kHz：~300MB
- 转码为 16kHz 单声道 MP3：~50-80MB
- 转码为 16kHz 单声道 WAV：~200MB

**从 2GB 降到 50-200MB，数据量减少 90-97%。**

## 三、方案对比

### 方案 A：前端 FFmpeg.wasm 提取音频（数据量最优）

```
用户选择 2GB 电影
        ↓
【前端】FFmpeg.wasm 在浏览器内提取音频轨道
        ↓
2GB → 200MB 音频 (AAC) → 可选重采样到 16kHz 单声道 → ~60MB
        ↓
【网络】只上传 60MB 音频
        ↓
【后端】直接转录 60MB 音频
```

| 优点 | 缺点 |
|------|------|
| 数据量减少 90-97% | FFmpeg.wasm 包体约 25MB，首次加载慢 |
| 上传时间从分钟降到秒级 | 浏览器内存占用仍较大（需加载文件到 WASM） |
| 后端无需改动 | 老旧浏览器/设备可能不支持 WASM |
| 用户感知最好 | 集成复杂度中等 |

**适用场景**：网络较慢、文件极大、追求极致体验

---

### 方案 B：后端 Streaming + FFmpeg 实时转码（工程最简）

```
用户选择 2GB 电影
        ↓
【前端】正常上传整个文件（流式）
        ↓
【后端】UploadFile 异步迭代器逐块读取（不一次性入内存）
        ↓
边接收边写入磁盘 → 接收完成后 FFmpeg 提取音频 → 丢弃视频数据
        ↓
只把音频传给 ASR
```

| 优点 | 缺点 |
|------|------|
| 前端完全无需改动 | 仍然要传 2GB 数据（网络瓶颈未解决） |
| 后端内存占用恒定（不随文件大小增长） | 上传时间长（几 GB 需几分钟） |
| 去掉文件大小限制即可支持任意大小 | 本机回环网络仍占满带宽 |
| 实现复杂度低 | |

**适用场景**：快速落地、后端可控、网络环境好

---

### 方案 C：后端直接读取本地文件路径（理论最优，但受限）

```
用户选择 2GB 电影
        ↓
【前端】获取文件的绝对路径（C:\Movies\xxx.mkv）
        ↓
【网络】只传路径字符串（几十字节）
        ↓
【后端】直接读取本地文件 → FFmpeg 提取音频 → ASR
```

| 优点 | 缺点 |
|------|------|
| 零数据传输开销 | **浏览器安全限制：无法获取文件真实路径** |
| 极致速度 | 仅桌面端应用（Electron/Tauri）可实现 |
| 内存占用最小 | |

**适用场景**：如果未来打包为桌面应用（Electron），这是最佳方案

---

### 方案 D：用户手动预处理 + 上传音频（最轻量）

提示用户先用 FFmpeg 自行提取音频：
```bash
ffmpeg -i movie.mkv -vn -acodec libmp3lame -ac 1 -ar 16000 -b:a 64k movie_audio.mp3
```

| 优点 | 缺点 |
|------|------|
| 零开发成本 | 用户门槛高 |
| 数据量最小 | 不友好 |
| | 无法自动获取字幕时间轴 |

## 四、推荐方案：B + A 渐进式组合

### Phase 1（立即做）：后端 Streaming + 音频提取

**目标**：先解决"不能传大文件"的问题，前端无需改动。

改动点：
1. 后端 `/api/transcribe` 改为 Streaming 接收，去掉 `MAX_UPLOAD_MB` 硬限制
2. 不一次性 `await file.read()`，改用 `UploadFile.file` 异步迭代逐块读取
3. 边接收边写入磁盘，接收完成后用 FFmpeg 提取音频轨道
4. 只把音频传给 ASR
5. 添加进度日志（每 100MB 输出一次）

预期效果：
- 支持任意大小文件上传
- 后端内存占用不再随文件大小增长
- 但仍需等待整个文件上传完成

### Phase 2（本周）：前端本地音频提取

**目标**：解决"传 2GB 视频太蠢"的问题。

调研两个方向：
1. **FFmpeg.wasm**：浏览器内提取音频，只传音频（减少 90% 数据量）
2. **WebCodecs API**（Chrome 94+）：浏览器原生解码视频，提取音频轨道

预期效果：
- 2GB 电影 → 只传 60-200MB 音频
- 上传时间从 5 分钟降到 30 秒

### Phase 3（未来）：桌面端直接路径读取

如果未来用 Electron/Tauri 打包：
- 前端直接获取文件绝对路径
- 后端直接读取本地文件
- 零网络传输开销

## 五、Phase 1 详细设计

### 5.1 后端改造：Streaming 接收 + FFmpeg 音频提取

当前代码问题（`main.py:364-377`）：
```python
file_bytes = await file.read()          # 2GB 读入内存！
# ...
tmp_path.write_bytes(file_bytes)        # 2GB 写入磁盘
# ...
await transcribe_audio(file_bytes, ...) # 2GB 传给 ASR
```

改造后流程：
```python
# 1. 流式接收文件到临时位置
chunk_size = 8192
with open(tmp_video_path, "wb") as f:
    while chunk := await file.read(chunk_size):
        f.write(chunk)                  # 每次只处理 8KB，内存恒定

# 2. FFmpeg 提取音频（丢弃视频）
ffmpeg -i tmp_video.mp4 -vn -acodec libmp3lame -ac 1 -ar 16000 -b:a 64k tmp_audio.mp3

# 3. 只把音频传给 ASR（可能只有 50MB）
audio_bytes = tmp_audio_path.read_bytes()
await transcribe_audio(audio_bytes, "audio.mp3", "audio/mpeg", language)
```

### 5.2 前端改造：支持分片上传（断点续传）

对于几 GB 的文件，单请求上传存在风险：
- 网络中断需要从头再来
- 浏览器/代理超时

分片上传方案：
```javascript
const CHUNK_SIZE = 10 * 1024 * 1024; // 10MB/片

// 1. 初始化上传，获取 upload_id
const { upload_id, chunks } = await fetch('/api/upload/init', {
  method: 'POST',
  body: JSON.stringify({ filename, size, chunks: Math.ceil(file.size / CHUNK_SIZE) })
});

// 2. 逐片上传
for (let i = 0; i < chunks; i++) {
  const chunk = file.slice(i * CHUNK_SIZE, (i + 1) * CHUNK_SIZE);
  await fetch(`/api/upload/chunk/${upload_id}/${i}`, {
    method: 'POST',
    body: chunk,
  });
}

// 3. 合并并触发转录
await fetch(`/api/upload/finish/${upload_id}`, { method: 'POST' });
```

但分片上传的工程复杂度较高，对于本地回环网络，**单请求流式上传通常已足够**（回环网络速度 > 1GB/s）。

## 六、关键决策

| 决策 | 建议 | 理由 |
|------|------|------|
| 是否去掉 200MB 限制？ | **是** | 本地应用不应有比用户磁盘更小的限制 |
| 是否做分片上传？ | **先不做** | 本机回环网络极快，单请求足够；如用户反馈超时再做 |
| 是否集成 FFmpeg.wasm？ | **Phase 2 做** | 效果显著但集成成本高，Phase 1 先保底 |
| 是否支持桌面端直接路径？ | **未来做** | 需要 Electron/Tauri 改造，不在当前 scope |

## 九、代码层面的关键瓶颈点

### 瓶颈 1：`main.py:364` 一次性读入内存
```python
file_bytes = await file.read()  # 2GB 直接读入 Python 内存！
```

### 瓶颈 2：`main.py:377` 再次写入磁盘
```python
tmp_path.write_bytes(file_bytes)  # 2GB 写入磁盘
```

### 瓶颈 3：`main.py:381` 传给 ASR 时又是一次内存拷贝
```python
asr_result = await transcribe_audio(file_bytes, ...)  # 2GB 作为参数传递
```

### 瓶颈 4：`asr.py:155` ASR 内部再写一次临时文件
```python
with tempfile.NamedTemporaryFile(...) as src:
    src.write(file_bytes)  # 2GB 又写入一次临时文件
```

**总计：2GB 文件被读了 4 次、写了 3 次，内存中同时驻留 2GB。**

---

**结论**：Phase 1（后端 Streaming + FFmpeg 音频提取）是最务实的下一步，可以在不改动前端的情况下立即支持大文件，且后端内存占用不再随文件大小增长。Phase 2（前端 FFmpeg.wasm）是体验优化的下一步。
